Un proyecto o aplicación web generalmente se prueba manualmente. En el caso de metodologías ágiles necesariamente se deben aplicar pruebas unitarias y pruebas de aceptación del usuario. Esta corrida manual de las pruebas toma mucho tiempo por parte del equipo. En un principio se corrían todas fallaban las demás. mano y si fallaba una ya

El resultado del trabajo de este equipo es una solución propuesta para realizar integración continua:
1. Instalar PHP, Apache... Java JDK y JRE y Mercurial HG.
2. PEAR.  iNSTALAR php uNIT, CODE SNIFER, php Documentor.
3. php

Una integración continua como está permite generar ciertas métricas. 

De las 8000 líneas que se han ejecutado el equipo de desarrollo ha generado solamente 4000... Existe la necesidad de identificar estas líneas.

Experiencias y conclusiones
1) Cada cambio en la Base de Datos e información cambian pueden generar errores que no son errores. Posible alternativa es mantener una base de datos correctas.
2) No se ha establecido el orden en que deben ejecutarse las pruebas. Las pruebas se ralizan de una manera no determinada este momento.
3) Se debe de hacer todo un caso de uso en una sola prueba. 
4) Antes las frecuencia de las pruebas eran cada dos días. Ahora en cada comit se ejecutan todas las pruebas.

Lecciones aprendidas:
1) Crear el script de datos de pruebas para realizar las pruebas.
2) Se requiere una sola persona para ejecutar las pruebas.

Observaciones

  • Debe existir alguna manera de evaluar solamente o de contar solamente las líneas desarrolladas por el equipo, sin tomar en cuenta las pruebas generadas por el framework.
  • Para la prueba de violación al estándar de código se deben de identificar también las líneas generadas por el equipo de trabajo.
  • Código muerto por definición según wikipedia es que se ejecuta pero los resultados no sirven para nada. El Dr. Lemus menciona que el código muerte es el que no se ejecutó.
